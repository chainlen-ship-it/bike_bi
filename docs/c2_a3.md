# A3 Thinking — Cyclistic Course 2: Data Preparation

**Goal:** Combine bike, weather, zip-code, and boundary data into materialized reporting tables that feed both an executive overview and an analyst drill-down on the Cyclistic dashboard.

---

## Why This Stage Exists

Course 1 produced 9 ad-hoc SQL queries. They answer analytical questions but they are not shareable artifacts — they return result sets on demand, not persistent tables that a dashboard or a non-engineer stakeholder can use directly.

Course 2 is the data preparation stage: take the analytical thinking from Course 1 and turn it into a reliable, repeatable data layer. Two new sources arrive that Course 1 didn't have: daily weather and geographic enrichment. Both were gaps that limited what the dashboard could answer.

**The weather gap:** Course 1 could show that ridership drops in winter. It couldn't show whether that drop is weather-driven or just seasonal. Joining NOAA GSOD daily precipitation and temperature data lets the dashboard answer "does ridership drop specifically on rainy days?" rather than just "does ridership drop in cold months?" Those are different operational questions with different implications for fleet planning.

**The geography gap:** Trip data has latitude and longitude coordinates. Without borough and neighborhood labels, the map is a set of dots that executives can't interpret. The zip code join turns coordinates into names. Sara Romero and Ernest Cox can now read "Lower East Side" instead of "40.7148, -73.9818."

## 1. Background

Course 1 produced 9 ad-hoc SQL queries against NYC Citi Bike trip data and the Course 1 planning documents. Course 2 is the **Data Preparation** stage of the BI workflow: extract, transform, and combine the data into target reporting tables that can be loaded into Tableau or read by the existing HTML dashboard.

Two new sources arrive in Course 2:

- **NOAA GSOD** daily weather (`bigquery-public-data.noaa_gsod.gsod20*`), filtered to wban `94728` (Central Park) as a city-wide proxy
- An **NYC zip-code CSV** that maps zip → neighborhood + borough (loaded as `cyclistic.zip_codes` in the GCP project)

A third dataset, `bigquery-public-data.geo_us_boundaries.zip_codes`, is used for spatial joins (`ST_WITHIN`) between trip start/end coordinates and zip polygons.

## 3. Goals / Targets

Two materialized target tables in BigQuery:

- `reporting_full_year` — all of 2014–2015, enriched with weather and geography
- `reporting_summer` — July, August, September of 2014–2015 for seasonal focus

Each row enriched with:
- usertype (Subscriber / Customer)
- start zip / borough / neighborhood
- end zip / borough / neighborhood
- start_day / stop_day (DATE_ADD +5 years applied for the fictional-project "recent" feel — removable for true historical view)
- daily mean temperature, mean wind speed, total precipitation
- 10-minute trip-duration bucket (reduces row count without losing distribution shape)
- trip_count

**Why pre-aggregate to 10-minute buckets:**
The raw data has one row per trip. At millions of rows, dashboard rendering slows significantly. Bucketing trip duration to the nearest 10 minutes collapses near-duplicate rows while preserving the shape of the duration distribution. This is the pre-aggregation principle applied in practice.

**Why two tables instead of one with a filter:**
The summer table is queried frequently and independently — it supports a dedicated dashboard view. Materializing it separately means the full-year table isn't scanned every time a stakeholder opens the summer borough chart. Separate tables is the right performance trade-off here.

Both tables are Tableau-ready and CSV-export-ready so the same data layer feeds the HTML dashboard and any future Tableau workbook.

## 4. Analysis Questions Added in Course 2

Building on Course 1 questions:

- How does daily precipitation impact trip volume per borough?
- Do summer (Jul–Sep) peaks differ across boroughs and neighborhoods?
- Do subscriber vs casual patterns diverge in inclement weather — and by how much?
- Is the weather sensitivity different for longer trips vs shorter trips?
- Which boroughs are growing fastest year-over-year?

**The weather assumption worth flagging:**
NOAA GSOD records precipitation totals per day, not per hour. A day with 0.3 inches of rain at 3am looks identical to a day with 0.3 inches at 5pm rush hour. For this dashboard, any precipitation on a trip day is treated as potentially impacting ridership. That's a simplification but a documented one — it's labeled as an assumption in the dashboard.

## Notes / Conventions Adopted from the Course Query

- `DATE_ADD(..., INTERVAL 5 YEAR)` shifts dates forward so the dashboard appears recent. This is a fictional-project convention — remove for a true historical view.
- `ROUND(CAST(tripduration / 60 AS INT64), -1)` buckets trip durations to the nearest 10 minutes to reduce row count without losing distribution shape.
- `WEA.wban = '94728'` — Central Park weather station used as a city-wide proxy. Limitation: precipitation intensity and timing vary across the five boroughs.
